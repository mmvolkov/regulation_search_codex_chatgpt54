# Обследование проекта «Помощник для сотрудников»

## Что это за проект

**Помощник для сотрудников** — это корпоративная поисковая система по внутренним регламентам компании в формате `.docx`. Суть: сотрудник задает вопрос на естественном языке, система находит релевантные фрагменты в регламентах и формирует краткий ответ с цитатами-источниками.

Ветка `codex/logging-and-fallback-fixes` добавляет к базовому MVP систему логирования всех действий в Google Spreadsheet и фиксит fallback-поведение при отсутствии ответа.

---

## Архитектура (4 слоя)

### 1. Слой данных и парсинга (Python)

- `src/regulation_search/docx_parser.py` — ядро проекта. Читает Word-файлы через OOXML (zipfile + ElementTree), распознает заголовки (через Word-стили и нумерацию типа `2.1.4`), нормализует текст, строит иерархический `section_path`, выделяет три типа блоков: `paragraph`, `important`, `table_row`. Отдельно обрабатывает таблицы с merge-ячейками (вертикальные объединения, grid-span). Длинные фрагменты режет на chunks по предложениям (лимит ~1200 символов).
- `src/regulation_search/config.py` — dataclass `Settings`, который собирает конфигурацию из `.env`: URL Qdrant, ключ OpenAI, модель эмбеддингов (`text-embedding-3-large`), параметры chunking. Модель генерации ответа выбирается пользователем в UI из двух вариантов: `GPT-4o mini` (по умолчанию) и `GPT OSS 120B` (экспериментальная). Выбор передаётся через frontend в поле `model` и маршрутизируется через dispatcher в `search-api`.

### 2. Слой индексации (Python + Qdrant)

- `src/regulation_search/qdrant_indexer.py` — создает **гибридный индекс** в Qdrant:
  - **dense** embeddings через OpenAI `text-embedding-3-large` (семантический поиск по смыслу);
  - **sparse** embeddings через `fastembed` модель `Qdrant/bm25` (точный поиск по терминам).
  - Коллекция `regulations_hybrid` с payload-индексами по `doc_id`, `doc_title`, `block_type`, `source_file`.
- `scripts/parse_documents.py` и `scripts/index_documents.py` — CLI-утилиты для локального парсинга и индексации корпуса.
- `src/regulation_search/ingest_api.py` — FastAPI-микросервис для docker-окружения: `GET /health`, `POST /upload`, `POST /ingest`.

### 3. Слой оркестрации (n8n dispatcher)

- `n8n/regulation_search_dispatcher.json` — **главный** production workflow (~2270 строк JSON). Это n8n-диспетчер, принимающий webhook по `/webhook/regulation-search-dispatch` и маршрутизирующий по полю `action`:
  - `authorize` — проверка доступа по allowlist из Google Sheets (лист `authorized_users`);
  - `search` — проксирование поиска на `search-api`, сбор ответа;
  - `upload` — загрузка и индексация DOCX;
  - `collection_status` / `collection_clear` — управление коллекцией Qdrant;
  - `feedback` — сохранение оценки полезности ответа.
  - **Логирование**: каждое действие записывается в Google Spreadsheet по листам `auth_log`, `interaction_log`, `feedback_log`, `document_loading_log`.
- `n8n/regulation_search_hybrid.json` — reference-реализация гибридного поиска как standalone n8n workflow.

### 4. Слой интерфейса (HTML/JS/CSS + PHP proxy)

- `site/index.html` + `site/app.js` + `site/styles.css` — SPA-подобный frontend «Помощник для сотрудников» с двумя режимами:
  - **Поиск**: ввод вопроса, выбор модели (GPT-4o mini / GPT OSS 120B), отображение ответа, подтверждающих фрагментов с рангом и релевантностью, feedback "Да/Нет", история запросов.
  - **Загрузка документов**: drag-and-drop DOCX, индексация, просмотр статуса коллекции Qdrant (количество фрагментов), очистка коллекции.
  - Логика определения fallback-ответа (`isFallbackAnswerText`) — проверяет на паттерны типа "к сожалению", "не найдена информация" и т.д.
- `site/regulation-proxy/` — набор PHP-прокси (`search.php`, `upload.php`, `auth.php`, `feedback.php`, `collection.php`), которые решают проблему CORS и скрывают внутренние URL. Все проксируют запросы на один upstream URL: `https://plequeneluera.beget.app/webhook/regulation-search-dispatch` (это n8n dispatcher).
- `site/regulation-proxy/access.php` — shared-логика: URL диспетчера, forwarded-заголовки, локальный allowlist пользователей с ролями (admin/editor/viewer).

## Production-стек на Beget

`deploy/beget/docker-compose.yml` описывает полный стек:

| Сервис | Роль |
|---|---|
| **Traefik** | HTTPS, routing, Let's Encrypt |
| **n8n** + **n8n-worker** | Оркестрация, webhook dispatcher |
| **Postgres** | Хранилище для n8n |
| **Redis** | Очереди для n8n worker |
| **Qdrant** v1.14.1 | Векторная БД |
| **ingest-api** | FastAPI для загрузки DOCX |
| **regulation-search-api** | Backend поиска (собирается из отдельного репозитория!) |

Локально используется только Qdrant v1.16.3 (корневой `docker-compose.yml`).

## Что характерно для ветки `codex/logging-and-fallback-fixes`

1. **Структурированное логирование** — 4 листа в Google Spreadsheet (`auth_log`, `interaction_log`, `feedback_log`, `document_loading_log`) с детальными схемами. Описано в `docs/LOGGING_SCHEMA.md`.
2. **Feedback flow** — пользователь может оценить ответ "Да/Нет", данные уходят через `feedback.php` → dispatcher → Google Sheets.
3. **Fallback-определение** — фронтенд распознает "пустые" ответы по паттернам (`fallbackAnswerPatterns`) и полям `answerFound` / `responseType`, чтобы корректно показывать UI.
4. **Deploy bundle** — `scripts/build_logging_deploy_bundle.sh` собирает архив для быстрого деплоя.
5. **Документация деплоя** — `LOGGING_DEPLOY_RUNBOOK.md` и `LOGGING_SMOKE_TEST_CHECKLIST.md` с пошаговыми инструкциями.

## Ключевые технические особенности

- **Парсер DOCX** собственный, работает напрямую с OOXML — не через `python-docx`, а через `zipfile` + `ElementTree`. Это дает полный контроль над таблицами с merge-ячейками и нестандартными стилями.
- **Гибридный поиск** (dense + sparse) реализован нативно через Qdrant — это лучше работает по регламентам, где важны и точные формулировки, и семантическая близость.
- **n8n выступает как middleware/orchestrator**, а не как единственный backend. Реальный `search-api` — отдельный сервис.
- **PHP proxy** — прагматичное решение для same-origin доступа на shared-хостинге Beget.

Проект находится на стадии рабочего MVP с полным циклом: парсинг → индексация → поиск → ответ → feedback → логирование.

## Схема потоков данных

### Поисковый поток

```
Пользователь
  → Frontend (site/app.js)
    → PHP proxy (search.php)
      → n8n dispatcher (authorize + search)
        → search-api
          → OpenAI (dense embedding)
          → Qdrant (hybrid search)
        ← fragments + answer
      ← JSON
    ← JSON
  ← Ответ + источник + фрагменты
```

### Поток индексации

```
Администратор
  → Frontend (site/app.js, вкладка «Загрузка»)
    → PHP proxy (upload.php)
      → n8n dispatcher (authorize + upload)
        → ingest-api / search-api
          → DOCX parser → chunks
          → OpenAI (dense embeddings)
          → Qdrant (upsert)
        ← stats
      ← JSON
    ← JSON
  ← Статус индексации + обновленная коллекция
```
