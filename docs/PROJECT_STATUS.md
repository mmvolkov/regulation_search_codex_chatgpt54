## Чанкование (chunker.py) — детальный статус

| # | Требование | Статус | Пояснение |
|---|---|---|---|
| 1 | Из документа выделяются текстовые абзацы, таблицы и заголовки | ✅ | Итерация по `<w:p>` и `<w:tbl>` элементам body |
| 2 | Исключаются абзацы со стилем TOC | ✅ | `_is_toc_style()` — проверяет `"toc"` и `"содержание"` |
| 3 | Заголовки по стилю «Заголовок» или весь абзац жирным | ✅ | `_is_heading()` = `_is_heading_style()` + `_is_all_bold()` |
| 4 | Таблицы 1x1 → текстовый абзац | ✅ | `chunker.py:268-276` |
| 5 | Таблицы иного размера → markdown с объединёнными ячейками | ✅ | `_table_to_markdown()` + `_get_merged_cell_text()` обрабатывает `gridSpan` |
| 6 | Таблицы в markdown → фрагменты 1:1 | ✅ | Таблицы сразу добавляются в результат, не группируются |
| 7 | Удаление служебных отметок `[Новое]` | ✅ | `_clean_text()` — `re.sub(r'\[(?:Н|н)овое\]', '', ...)` |
| 8 | Объединение маленьких абзацев в один фрагмент (общий заголовок) | ✅ | `_merge_small_chunks()` — группирует по heading, объединяет если < `min_chars` |
| 9 | Крупные абзацы делятся по предложениям | ⚠️ Частично | `_split_large_text()` работает, **но используется простой regex, а не Natasha Razdel** |
| 9a | Для выделения предложений используется Natasha Razdel | ❌ | Используется `_split_sentences_simple()` — regex. Комментарий в коде: *"Simple sentence splitter (Stage 1 - without Natasha)"* |
| 10 | Средние абзацы → фрагменты 1:1 | ✅ | Если не мелкий и не крупный, проходит как есть |
| 11 | К фрагменту добавляется ближайший заголовок (последний при нескольких подряд) | ✅ | `current_heading` перезаписывается при каждом заголовке → сохраняется только последний |

**Итог по чанкованию**: 10 из 11 пунктов реализованы. Единственный пробел — Natasha Razdel для выделения предложений (используется простой regex-сплиттер).

## Лемматизация (lemmatizer.py) — детальный статус

| # | Требование | Статус | Пояснение |
|---|---|---|---|
| 1 | Текст разбивается на предложения с помощью Natasha | ❌ | Разбиение на предложения не используется — текст сразу токенизируется |
| 2 | Слова классифицируются с помощью Natasha (POS-tagging) | ❌ | Нет классификации слов. Стоп-слова — жёсткий статический список |
| 3 | Из списка исключаются стоп-слова на основе классификации | ⚠️ Частично | Стоп-слова удаляются, но **по статическому списку**, а не по POS-классификации Natasha |
| 4 | Слова приводятся к базовой форме с помощью Natasha | ❌ | **Лемматизации нет**. Слова остаются в исходной форме. Комментарий: *"Simple lemmatizer for Stage 1 (without Natasha)"* |

**Итог по лемматизации**: Реализована только заглушка Stage 1. Из 4 пунктов ни один не использует Natasha. Код явно помечен как "Stage 1" — упрощённая реализация без зависимости от Natasha. Stage 2 так и не был реализован.

**Влияние на качество**: Отсутствие лемматизации снижает качество sparse-поиска (BM25). Например, запрос «переназначение» не найдёт фрагмент со словом «переназначить», потому что слова не приведены к общей базовой форме.

## Итог

Базовый пайплайн реализован и развёрнут в production:

**Чанкование → Токенизация (без лемматизации) → Гибридный поиск (dense + sparse + RRF) → Проверка релевантности → Генерация ответа**

Дополнительно реализованы: выбор модели, режимы полноты ответа (S/M/L), логирование, авторизация, обратная связь, корректная обработка случая «ответ не найден».

**Ключевой технический долг**: лемматизация (Stage 2 с Natasha) не реализована — снижает качество sparse-поиска. Dense-поиск (embeddings) компенсирует это частично, но не полностью.

Backlog из 20 улучшений — идеи на будущее из исходного документа описания прототипа.
